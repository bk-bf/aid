# BUG-007: dotfiles git repo deletes ~/.config/nvim-tdl symlink on branch operations

**Status**: open  
**Branch**: master (`install.sh`, `aid.sh`)  
**First appeared**: 2026-03  

---

## Symptoms

- After a branch switch, pull, or reset in any git repo that tracks `~/.config/`, the
  `~/.config/nvim-tdl` symlink silently disappears or is replaced by whatever that repo
  tracked at that path.
- `aid` launches and nvim starts, but loads zero config — no plugins, no keymaps, no
  `aidignore.lua`. Indistinguishable from a first-run before install.
- Restarting the aid session repeatedly does not help (the symlink stays gone).
- A Lua module cache bug is the natural first suspect, wasting significant debug time.

## Root cause

`install.sh` creates `~/.config/nvim-tdl` as a symlink:

```
~/.config/nvim-tdl -> <aid-install-dir>/nvim/
```

`~/.config` is a common location for dotfiles repos (e.g. `hyprland-noctalia`,
`chezmoi`, `GNU stow`). If that repo ever tracked `nvim-tdl` — even as a stale entry
from a previous install method — git stores it as a symlink object. On any
`checkout`, `pull`, `reset --hard`, or `merge`, git restores that object, which means
**deleting the current symlink and writing its own version** (or simply deleting it if
the checked-out ref has no entry for that path).

The net result: a routine dotfiles maintenance operation silently breaks aid with no
error message and no obvious connection to the action that caused it.

## Why it is hard to diagnose

Both of these produce the identical symptom ("my code changes aren't taking effect"):

1. This bug — nvim never loaded your config at all.
2. Lua module caching — nvim loaded a stale cached version of a Lua module.

Because they look the same, the natural response (restart the session) correctly fixes
case 2 but has zero effect on case 1 — leading to repeated restarts and no progress.

### Decisive diagnostic (run this first, before anything else)

```bash
# Is the symlink alive?
ls -la ~/.config/nvim-tdl

# Is nvim actually loading YOUR config?
NVIM_APPNAME=nvim-tdl nvim --headless \
  -c "lua print(vim.fn.stdpath('config'))" \
  -c q 2>&1
```

If the second command prints anything other than `<aid-install-dir>/nvim`, the
problem is infrastructure, not code. Stop debugging Lua.

### Workaround (immediate)

```bash
ln -sfn <aid-install-dir>/nvim ~/.config/nvim-tdl
echo '/nvim-tdl' >> ~/.config/.gitignore
echo '/nvim'     >> ~/.config/.gitignore
```

## Proposed fix: eliminate the symlink

The symlink only exists because nvim resolves config as
`$XDG_CONFIG_HOME/$NVIM_APPNAME`. If `XDG_CONFIG_HOME` is set to the aid install
dir's parent at nvim launch time, no symlink is needed:

```bash
# In aid.sh, when spawning nvim:
XDG_CONFIG_HOME="$TDL_DIR/.." NVIM_APPNAME=nvim nvim ...
# nvim resolves config to: $TDL_DIR/../nvim == main/nvim  ✓
```

Full implications:

| Concern | Impact |
|---|---|
| Config dir | `$TDL_DIR/nvim` — already the real location, no change |
| Data dir (`lazy` cache, sessions, etc.) | Still `~/.local/share/nvim-tdl` — unchanged because `XDG_DATA_HOME` is not overridden |
| `~/.config` | Untouched. Users can track it however they want |
| `install.sh` step 4 | Symlink creation becomes unnecessary, can be removed |
| nvim-treemux | Same fix applies: `XDG_CONFIG_HOME="$TDL_DIR/.." NVIM_APPNAME=nvim-treemux nvim ...` |

To also isolate the data and state dirs (plugin cache, shada, etc.) from the user's
main nvim, set all three XDG vars explicitly:

```bash
XDG_CONFIG_HOME="$TDL_DIR/.." \
XDG_DATA_HOME="$HOME/.local/share/aid" \
XDG_STATE_HOME="$HOME/.local/state/aid" \
NVIM_APPNAME=nvim nvim ...
```

This gives complete isolation without any symlink in `~/.config`.

## Affected users

Anyone who:
- manages dotfiles with git, chezmoi, GNU stow, or similar in `~/.config/`, **and**
- has ever had a `nvim-tdl` entry in that repo (e.g. from a previous install), **or**
- tracks all of `~/.config/` and the repo does not explicitly ignore `nvim-tdl`

This is a large fraction of the Arch/NixOS/hyprland target audience — these users
almost universally use dotfile managers.
