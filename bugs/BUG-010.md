# BUG-010: opening an already-open file creates a duplicate tab

**Status**: open  
**Branch**: master (`nvim-treemux/treemux_init.lua`)  
**First appeared**: 2026-03  

---

## Symptoms

- Open a file from the sidebar tree (`l` / `<CR>`).
- Open the same file again from the sidebar tree.
- The tab bar now shows two tabs for the same file instead of focusing the existing one.
- The buffer is loaded twice; both tabs display identical content.

## Root cause

The open path in the sidebar nvim is:

```
treemux_init.lua:tabnew_follow_symlinks()
  → nvim_tree_remote.remote_nvim_open(socket_path, "tabnew", path, tmux_opts)
    → transport.open(path, "tabnew", socket_path, 0)
      → transport.exec("tabnew <path>", socket_path, 0)
        → vim.rpcrequest(chan, "nvim_command", "tabnew <path>")
```

`transport.open()` (`nvim-tree-remote.nvim/lua/nvim_tree_remote/transport.lua:76–81`) constructs
a raw Ex command string (`"tabnew <path>"`) and fires it into the main nvim over msgpack-RPC.
Vim's `:tabnew` unconditionally opens a new buffer — it has no awareness of whether the target
path is already loaded. No deduplication check exists anywhere in the call chain.

The `duplicate_*` highlight groups defined in the bufferline config (`nvim/init.lua:429–431`) are
intended for files that share a filename across different directories — not for the same file
loaded twice. Their presence does not indicate intentional duplicate-tab support.

## Proposed fix

The fix belongs in `tabnew_follow_symlinks()` in
`main/nvim-treemux/treemux_init.lua:53–63`.

Before issuing `tabnew`, use `transport.exec` to RPC into the main nvim and check whether the
target path is already a loaded buffer:

```lua
-- check if path is already loaded in the main nvim
local bufnr_result = nil
pcall(function()
  local chan = vim.fn.sockconnect("pipe", socket_path, { rpc = true })
  bufnr_result = vim.rpcrequest(chan, "nvim_call_function", "bufnr",
    { vim.fn.fnamemodify(node.absolute_path, ":p") })
  vim.fn.chanclose(chan)
end)

if bufnr_result and bufnr_result ~= -1 then
  -- buffer already open — switch to it instead of opening a new tab
  require("nvim_tree_remote.transport").exec(
    "buffer " .. bufnr_result, socket_path, 0)
else
  -- not open yet — proceed with tabnew as normal
  require("nvim_tree_remote").remote_nvim_open(socket_path, "tabnew", node.absolute_path, tmux_opts)
end
```

`bufnr(<path>)` returns `-1` when the path is not loaded, or a positive integer buffer number
when it is. `buffer <n>` switches to the existing buffer without creating a new tab.

The existing `transport` module already provides `transport.exec` for the switch command, so no
new infrastructure is required.

### Edge cases to verify

- Path normalisation: `node.absolute_path` should already be absolute, but wrapping with
  `fnamemodify(..., ":p")` guards against trailing-slash differences.
- The `tabnew_main_pane` variant (`o` key, `nt_remote.tabnew_main_pane`) uses a different code
  path and may need the same treatment.
- The `neo-tree` `file_open_requested` hook in `treemux_init.lua:347–368` routes through the same
  `remote_nvim_open` call and would also benefit from the same guard.
