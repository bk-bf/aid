# BUG-021: LSP/treesitter semantic tokens bleed fg color over bufferline buffer name

**Status**: closed — fixed 2026-03-09  
**Branch**: main (`nvim/init.lua`)  
**First appeared**: recurring  
**Fixed**: 2026-03-09

---

## Symptoms

- The active (selected) tab in the bufferline shows the buffer filename in the wrong
  color — typically yellow or another LSP semantic token color — instead of the
  expected white (`p.tab_fg = "#ffffff"`).
- A grey/colored box appears around the buffer name in the selected tab.
- The color reverts to correct only after navigating away and back, or after a full
  reload.

## Root cause

Neovim's LSP semantic token highlighting (and treesitter) applies highlight groups such
as `@lsp.type.namespace`, `@lsp.type.variable`, `@variable`, etc. to the tabline buffer.
These groups carry a `fg` attribute that is **combined** with the base highlight group
(`BufferLineBufferSelected`) by default.

Neovim's highlight combination rules mean that a semantic token highlight with only `fg`
set (no `bg`) is merged on top of `BufferLineBufferSelected`, overriding the `fg` while
leaving the `bg` intact. This produces the visual symptom: the background is the correct
selected-tab color (`p.tab_sel`) but the text color is the LSP token color (e.g. yellow
for a Lua function name that matches the filename).

## Fix

Added `nocombine = true` to `BufferLineBufferSelected` and `BufferLineBufferVisible` in
both the bufferline `opts.highlights` table (applied at startup) and in `apply_palette()`
(applied at hot-reload):

```lua
buffer_selected = { fg = p.tab_fg, bg = p.tab_sel, bold = true, nocombine = true },
buffer_visible  = { fg = p.tab_fg, bg = p.tab_bg, nocombine = true },
```

`nocombine` instructs Neovim not to merge any other highlight attributes into this group,
making it immune to LSP/treesitter token bleeding.

| File | Change |
|------|--------|
| `main/nvim/init.lua` | `nocombine = true` on `buffer_selected` / `buffer_visible` in opts and `apply_palette()` |
